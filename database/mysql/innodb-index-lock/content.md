# InnoDB 인덱스와 잠금

> 이 글은 [Real MySQL 8.0](https://www.yes24.com/Product/Goods/103415627) 책을 참고하여 작성되었습니다.

- InnoDB에서의 잠금은 레코드를 잠그는 것이 아니라 인덱스의 레코드를 잠그는 방식으로 처리됨
- 즉, 변경해야 할 레코드를 찾기 위해 검색한 인덱스의 레코드 모두 락을 걸어야 함

### 예시
데이터베이스에 아래와 같이 설정된 `employees`라는 테이블이 존재한다고 가정하자.
- 해당 테이블에는 first_name 컬럼만 멤버로 하는 ix_firstname 인덱스가 준비되어 있다.
- `first_name='Georgi'`인 사원은 총 253명이다.
- `first_name='Georgi' and last_name='Klassen'`인 사원은 1명이다.

이때 아래와 같은 업데이트 쿼리를 실행할 경우 어떻게 락이 걸리는지 알아보자.
```mysql
UPDATE employess SET hire_date=NOW() WHERE first_name='Georgi' and last_name='Klassen';
```

위와 같은 업데이트 쿼리가 실행되면 1개의 레코드만 업데이트가 된다. 하지만 이 1건의 업데이트를 위해 총 253건의 레코드에 락이 걸리게 된다.

그 이유는 `last_name` 컬럼에 인덱스가 없기 때문에 `first_name='Georgi'`에 해당하는 레코드 253건이 전부 잠기게 되는 것이다.

따라서 업데이트를 위한 인덱스가 제대로 준비되어 있지 않다면 동시성이 상당히 떨어져서 한 세션에서 업데이트하는 중에는 다른 클라이언트는 업데이트를 하지 못하고 기다려야 하는 상황이 발생한다.

만약 `employees` 테이블에 총 레코드가 100만건이고, 인덱스가 하나도 없는 상황에서 위와 같은 업데이트 쿼리가 실행된다면 풀 테이블 스캔을 통해 100만건의 레코드가 전부 잠기게 된다.
