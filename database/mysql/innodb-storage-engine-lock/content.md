# InnoDB 스토리지 엔진 잠금

- MySQL에서 제공하는 잠금과는 별개로 스토리지 엔진 내부에서 레코드 기반의 잠금 방식 탑재
- 잠금 정보가 상당히 작은 공간으로 관리되기 때문에 페이지 락, 테이블 락으로 에스컬레이션이 되는 경우는 없다.

### 공유 및 배타적 락 (Shared and Exclusive Locks)
- InnoDB는 공유(S) 락, 배타적(X) 락이라는 두 종류의 행 수준(row-level) 잠금을 지원한다.
- S락은 락을 보유한 트랜잭션이 행을 읽을 수 있도록 허용
- X락은 락을 보유한 트랜잭션이 행을 업데이트하거나 삭제할 수 있도록 허용
- 만약 트랜잭션 T1이 행 r에 대해 S락을 보유하고 있다면, 행 r에 대한 S락을 요청하는 트랜잭션 T2의 요청은 다음과 같이 처리됨
  - T2에 의한 S락 요청은 즉시 허용. T1, T2 모두 행 r에 대한 S락 보유
  - T2에 의한 X락 요청은 즉시 허용되지 않음
- 만약 트랜잭션 T1이 행 r에 대해 X락을 보유하고 있다면
  - 별개의 트랜잭션 T2에게 어떠한 유형의 락도 즉시 승인될 수 없음
  - T2 트랜잭션은 T1이 행 r에 대한 락을 해제할 때까지 기다려야 함

### 레코드 락(Record lock, Record only lock)
- 다른 상용 DBMS와 동일하게 레코드 자체만을 잠그는 락
- InnoDB의 레코드 락은 레코드 자체가 아닌 `인덱스의 레코드를 잠그는 것`
  - 인덱스가 하나도 없는 테이블이더라도 내부적으로 자동 생성된 클러스터 인덱스를 이용해 잠금 설정

### 갭 락(Gap lock)
- 레코드 자체가 아니라 `레코드와 바로 인접한 레코드 사이의 간격만을 잠그는 것`을 의미
- 예시
  - `SELECT c1 FROM t WHERE c1 BETWEEN 10 and 20 FOR UPDATE;`
  - 위의 쿼리를 실행하는 경우 10, 20 사이의 갭이 모두 잠기므로 t.c1열에 15값을 삽입하는 다른 트랜잭션을 방지함
- 고유 인덱스를 사용하는 경우 갭 락이 사용되지 않음
  - `SELECT * FROM child WHERE id = 100;`
  - 이 경우 id값이 100인 행에 대한 인덱스 레코드 락만 사용하고 이전 갭에 값을 삽입하는 다른 세션은 고려하지 않음
- InnoDB의 갭 락은 다른 트랜잭션에서 갭에 값을 삽입하는 것을 방지하는 것
- 트랜잭션 격리 수준을 `READ COMMITTED`로 변경 시 해제 가능
  - 동시성을 높이기 위한 전략, 이 격리 수준에서는 넥스트 키 락 대신 레코드 락만 걸리게 됨

### 넥스트 키 락(Nest key lock)
- 레코드 락과 갭 락의 결합

### 자동 증가 락(Auto increment lock)
- 자동 증가 락은 `INSERT`, `REPLACE` 쿼리와 같이 새로운 레코드를 저장하는 쿼리에서만 필요
- UPDATE, DELETE 등의 쿼리에서는 걸리지 않음
- InnoDB의 다른 잠금과는 달리 트랜잭션에 상관없이 AUTO_INCREMENT 값을 가져오는 순간만 락이 걸렸다가 즉시 해제됨
- 자동 증가 락은 테이블에 하나만 존재하기 때문에 두 개의 `INSERT` 쿼리가 실행되는 경우 나머지 쿼리는 락을 기다려야 함
- 명시적으로 잠금을 획득하는 방법은 없으나, 아주 짧은 시간 잠겼다가 해제되기 때문에 대부분의 경우 문제가 되지 않는다.


- MySQL 5.1 버전 이상부터는 `innodb_autoinc_lock_mode`라는 시스템 변수를 이용해 작동 방식 변경 가능
- TRADITIONAL 모드 (0)
  - MySQL 5.0(위에서 설명)과 동일한 잠금 방식, 모든 INSERT문은 자동 잠금 락 사용
- CONSECUTIVE 모드 (1)
  - 한 건 또는 여러 건의 레코드를 생성하는 쿼리 중 MySQL에서 레코드 건수를 정확히 예측할 수 있는 경우에는 자동 증가 락 대신 훨씬 가볍고 빠른 뮤텍스를 사용함
  - `INSERT ... SELECT`와 같이 쿼리 실행 전 레코드 건수를 정확히 예측할 수 없는 경우는 자동 증가 락을 사용함
  - 할당받은 자동 증가 값이 남아서 사용되지 못하는 경우에는 연속되지 않고 누락된 값이 있을 수 있음 -> DBMS Sequence와 비슷한 맥락
- INTERLEAVED 모드 (2)
  - 절대 자동 증가 락 사용 X, 무조건 경량화된 래치(뮤텍스) 사용

> MySQL 5.7 버전까지는 기본값이 1, 8.0 버전부터는 기본값이 2로 바뀜
